# OWASP LLM Top 10

## What It Is

The OWASP Top 10 for Large Language Model Applications is a community-driven list of the ten most critical security risks when building and deploying applications powered by large language models. First published in 2023 and updated in 2025, it provides security practitioners and developers with a shared vocabulary for LLM-specific threats.

It is not a compliance framework — it is a threat taxonomy. Its value is in helping organisations understand what can go wrong with LLM-based systems and what controls to implement.

---

## Why It Matters

Traditional application security frameworks (OWASP Top 10 for web apps, SANS 25) were not designed for systems that generate natural language, follow instructions embedded in data, or operate autonomously. LLMs introduce a fundamentally different attack surface: the model itself can be manipulated through its inputs, and the model's outputs can contain attacks targeting downstream systems or users.

For any organisation deploying LLMs — whether commercial APIs, open-source models, or embedded AI features — the OWASP LLM Top 10 is the baseline threat model.

---

## The Top 10 Risks

### LLM01 — Prompt Injection

**Description:** An attacker crafts inputs that override the LLM's system prompt or intended behaviour, causing it to perform unintended actions, reveal sensitive information, or bypass safety controls.

Two forms:
- **Direct injection:** The attacker directly inputs malicious instructions to the model.
- **Indirect injection:** Malicious instructions are embedded in data the model processes (a document, a web page, an email) and execute when the model reads that data.

**Real-world impact:** A customer service bot instructed via injected text to reveal other customers' data. An agent browsing the web that executes instructions embedded in a malicious webpage. A RAG system that returns confidential documents when a retrieved chunk contains injection instructions.

**Controls:** Input validation and sanitisation; privilege separation between system and user instructions; output filtering; human review for high-stakes agentic actions; sandboxed execution environments.

---

### LLM02 — Insecure Output Handling

**Description:** The LLM's output is passed to downstream systems (browsers, databases, APIs, operating systems) without validation, enabling injection attacks in those systems.

**Real-world impact:** LLM output containing JavaScript injected into a web application causes XSS. SQL generated by an LLM and executed without parameterisation enables SQL injection. Shell commands generated by an LLM and executed without sandboxing allow system compromise.

**Controls:** Treat all LLM output as untrusted input to downstream systems. Apply standard injection prevention (parameterised queries, output encoding, command sandboxing) regardless of the source being an LLM.

---

### LLM03 — Training Data Poisoning

**Description:** An attacker corrupts the training data used to build or fine-tune a model, introducing backdoors, biases, or vulnerabilities that persist in the deployed model.

**Real-world impact:** A model fine-tuned on poisoned data consistently misclassifies a specific class of inputs (e.g., approving fraudulent transactions). A publicly scraped dataset contains adversarial examples designed to degrade model safety.

**Controls:** Data provenance controls; training data auditing; model behavioural testing against known poisoning signatures; preference for curated over scraped training data; supply chain controls for third-party datasets.

---

### LLM04 — Model Denial of Service

**Description:** An attacker crafts inputs that cause the LLM to consume disproportionate computational resources, degrading performance or causing service unavailability.

**Real-world impact:** Recursive or self-referential prompts that cause excessive token generation. Inputs designed to trigger worst-case inference paths. Context window flooding attacks.

**Controls:** Input length limits; rate limiting per user and session; timeout controls on inference; resource quotas; monitoring for abnormal token consumption patterns.

---

### LLM05 — Supply Chain Vulnerabilities

**Description:** Risks introduced through third-party models, datasets, plugins, or integrations used in the LLM application stack.

**Real-world impact:** A third-party LLM plugin with hidden data exfiltration. A pre-trained model from an untrusted source with embedded backdoors. A fine-tuning dataset containing copyright-infringing or poisoned content.

**Controls:** Model provenance verification; third-party plugin security assessment; software bill of materials (SBOM) for AI components; contractual security requirements for AI vendors; preference for models from established, audited providers.

---

### LLM06 — Sensitive Information Disclosure

**Description:** The LLM reveals sensitive information — system prompts, training data, personal data, or confidential business information — through its outputs.

**Real-world impact:** A model trained on proprietary data regurgitates that data when prompted. A system prompt containing API keys or internal architecture details is extracted by an attacker. A model reveals one user's personal information when prompted by another user.

**Controls:** Data minimisation in training; system prompt confidentiality (though note this is not a security boundary); output filtering for PII and sensitive data; access controls preventing cross-user data exposure; regular red-teaming for data extraction vulnerabilities.

---

### LLM07 — Insecure Plugin Design

**Description:** LLM plugins (tools, functions, integrations) are designed without adequate security controls, allowing the LLM to invoke them in unintended ways.

**Real-world impact:** A plugin with read/write access to a database is invoked by an injected prompt to exfiltrate or modify data. A plugin with broad API permissions is manipulated to perform actions beyond its intended scope.

**Controls:** Least privilege for plugin permissions; input validation within plugins (not relying on the LLM for validation); human confirmation for high-impact plugin actions; plugin scope documentation and review; OAuth scoping for API access.

---

### LLM08 — Excessive Agency

**Description:** The LLM is granted more autonomy, permissions, or capabilities than necessary, and acts beyond its intended scope — either through manipulation or design failure.

**Real-world impact:** An agent with email send permissions sends unsolicited messages on behalf of the user. An agent with file system access deletes or exfiltrates files. An agent with database access modifies production data based on misinterpreted instructions.

**Controls:** Principle of least privilege for agent capabilities; human-in-the-loop for irreversible or high-impact actions; clear scope boundaries defined in system prompts; monitoring and alerting for out-of-scope actions; audit trails for all agent actions.

---

### LLM09 — Overreliance

**Description:** Users or systems place excessive trust in LLM outputs without appropriate verification, leading to decisions based on hallucinated, biased, or incorrect information.

**Real-world impact:** A legal team relies on LLM-generated case citations that do not exist. A medical application provides treatment recommendations based on hallucinated drug interactions. A financial model uses LLM-generated market analysis containing fabricated data.

**Controls:** Clear communication to users about LLM limitations; mandatory human review for high-stakes outputs; grounding LLM outputs in verified sources (RAG with citation); output confidence indicators; prohibition on autonomous action based solely on LLM output in high-risk domains.

---

### LLM10 — Model Theft

**Description:** An attacker extracts the model's parameters, architecture, or training data through repeated querying — either to steal proprietary models or to develop more targeted attacks.

**Real-world impact:** A competitor extracts a fine-tuned proprietary model through model inversion attacks. An attacker uses extracted model knowledge to craft more effective adversarial examples.

**Controls:** Rate limiting and query monitoring; watermarking model outputs; detecting and blocking systematic extraction patterns; API authentication and access logging; legal protections (IP, terms of service) alongside technical controls.

## Relationship to Governance Frameworks

| Framework | Relationship |
|---|---|
| ISO 42001 | OWASP LLM Top 10 risks should be addressed through ISO 42001 Annex A controls, particularly in the AI system lifecycle and supply chain control categories. |
| NIST AI RMF | MAP function: OWASP Top 10 provides the threat taxonomy. MEASURE function: testing against Top 10 risks. MANAGE function: incident response for Top 10 attack classes. |
| EU AI Act | High-risk AI systems under the Act require technical robustness. OWASP controls directly support Article 15 (accuracy, robustness, cybersecurity) requirements. |

## Recommendations

1. Use the OWASP LLM Top 10 as the baseline threat model for every LLM deployment — it is the fastest way to identify security gaps specific to AI systems.
2. Prioritise LLM01 (Prompt Injection) and LLM08 (Excessive Agency) for agentic AI systems — these are the highest-impact risks in autonomous AI deployments.
3. Treat LLM output as untrusted in all downstream integrations (LLM02) — this is the most commonly missed control in application development.
4. For BFSI deployments, give particular attention to LLM06 (Sensitive Information Disclosure) and LLM09 (Overreliance) — these directly map to customer data protection and regulatory decision-making obligations.
5. Red-team LLM applications against the Top 10 before production deployment and on a recurring basis — models and attack techniques both evolve.
